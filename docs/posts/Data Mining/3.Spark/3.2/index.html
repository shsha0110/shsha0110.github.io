1. Introduction

  • 이전 포스트에서는 분산 컴퓨팅의 한계를 극복하기 위한 DAG(방향성 비순환 그래프) 모델의 도입과 Spark의 핵심 데이터 구조인 RDD(Resilient Distributed Dataset)의 개념, 그리고 지연 평가(Lazy Evaluation) 메커니즘을 살펴보았습니다.

  • 본 포스트는 그 후속편(Part 2)으로, Spark 내에서 RDD를 실제로 조작하고 변환하는 다양한 연산자(Operations)들을 수학적·논리적 관점에서 분석합니다. 더불어, 분산 시스템의 성능을 좌우하는 데이터 셔플링(Shuffling)과 관련된 핵심 개념인 Narrow Dependency와 Wide Dependency에 대해 심층적으로 다루어 보겠습니다.

2. 동기 부여 예제: 알파벳별 평균 단어 길이 계산

  • Spark 연산자들의 동작을 직관적으로 이해하기 위해, 분산 파일 시스템(DFS)에 저장된 대용량 텍스트 파일이 주어졌을 때 “각 알파벳(\(a, \dots, z\))으로 시작하는 단어들의 평균 길이를 구하는 문제”를 생각해보겠습니다.

2.1 MapReduce 방식의 접근

  • 기존 MapReduce 프레임워크를 사용한다면 이 문제는 다음과 같은 두 단계로 설계됩니다:
    • Map: 입력 텍스트의 각 라인을 읽어 단어로 분리한 뒤, 단어의 첫 글자와 그 길이를 Key-Value 쌍으로 매핑합니다.
      • (line index, line) \(\rightarrow\) {(w[0], len(w)) | w ∈ line}
    • Reduce: 같은 첫 글자(Key)를 가진 단어 길이들의 리스트를 모아 평균을 계산합니다.
      • (word_initial, lengths_list) \(\rightarrow\) (word_initial, mean(lengths_list))

2.2 Spark 방식의 RDD 파이프라인

  • Spark에서는 이 과정을 일련의 Transformation 체인으로 훨씬 유연하고 명확하게 작성할 수 있습니다.
avglens = sc.textFile(file) \
            .flatMap(lambda line: line.split()) \
            .map(lambda word: (word[0], len(word))) \
            .groupByKey() \
            .map(lambda kv: (kv[0], sum(kv[1]) / len(kv[1])))

3. 주요 RDD 연산자 (Operations) 상세 분석

  • Spark의 연산은 크게 새로운 RDD를 생성하는 Transformation과 실제 계산을 촉발하는 Action으로 나뉩니다. 앞선 예제에서 사용된 연산자들을 포함하여, 데이터 마이닝에서 필수적인 주요 연산자들을 정의해 보겠습니다.

3.1 FlatMap (Transformation)

  • 주어진 함수를 RDD의 모든 요소에 적용하는 변환 연산자입니다.
    • 논리적 구조: MapReduce의 Map 연산과 유사하지만 반환 타입에 제한이 없습니다. 하나의 입력 객체를 받아 0개 이상의 여러 객체들로 구성된 리스트를 반환할 수 있습니다.
    • Flattening: 반환된 여러 개의 리스트들은 최종적으로 중첩이 해제(flattened)되어 단일 RDD로 합쳐집니다. 텍스트 라인을 단어 단위로 쪼개는 작업(line.split())에 가장 적합합니다.

3.2 Map (Transformation)

  • FlatMap의 보다 엄격한(stricter) 버전입니다.
    • 논리적 구조: RDD의 각 입력 요소마다 정확히 단 1개의 객체를 반환해야 합니다. 리스트를 평탄화하는 flattening 작업이 수행되지 않습니다.
    • 이점: 시스템 입장에서 1:1 매핑이 보장되므로 리소스 관리와 파티션 크기 예측에 있어 훨씬 유리한 제어권을 갖습니다.

3.3 Filter (Transformation)

  • RDD의 객체 타입에 적용할 수 있는 조건식(Predicate)을 인자로 받습니다.
    • 해당 조건을 만족하는(참이 되는) 요소들만 남겨 새로운 RDD를 반환합니다.
    • 최적화 팁: 데이터의 크기를 줄여주기 때문에 파이프라인의 최대한 앞단(early stage)에 배치할수록 후속 연산의 부하(Network I/O 및 Memory)를 대폭 감소시킬 수 있습니다.
    • 예: .filter(lambda word: word not in stoplist)

3.4 Reduce (Action)

  • 지금까지의 Transformation과 달리, 최종적인 단일 값(Value)을 반환하는 Action 연산자입니다.
    • 함수 제약 조건: \((V, V) \Rightarrow V\) 타입의 함수를 인자로 받습니다. 즉, 동일한 타입의 두 값을 입력받아 하나의 값을 반환해야 합니다.
    • 수학적 성질: 분산 환경에서는 데이터가 어떤 순서로 묶여서 연산될지 알 수 없으므로, 넘겨지는 함수는 반드시 결합법칙(Associative)과 교환법칙(Commutative)을 만족해야 합니다. (예: 덧셈 연산 \(a + b\))
    • \((a + b) + c = a + (b + c)\) 및 \(a + b = b + a\)가 성립해야 분산 노드에서 각자 reduce한 결과를 다시 합칠 수 있습니다.

3.5 Join (Transformation)

  • 관계형 데이터베이스(RDB)의 조인과 같이 두 개의 Key-Value 쌍 RDD를 병합합니다.
    • 두 RDD는 동일한 타입의 Key를 가져야 합니다.
    • 특정 Key \(k\)에 대해 첫 번째 RDD에 \((k, x)\)가 있고 두 번째 RDD에 \((k, y)\)가 있다면, 조인 결과로 \((k, (x, y))\) 형태의 튜플을 생성합니다.
    • 각 RDD에 흩어져 있는 동일한 Key를 모아야 하므로 필연적으로 데이터를 네트워크를 통해 재분배해야 하며, 이는 Wide Dependency를 유발합니다.
x = sc.parallelize([("a", 1), ("b", 4)])
y = sc.parallelize([("a", 2), ("a", 3)])
x.join(y).collect() 
# 결과: [('a', (1, 2)), ('a', (1, 3))]

3.6 GroupByKey (Transformation)

  • MapReduce의 group-by-keys 단계와 동일한 역할을 수행합니다.
    • Key-Value 쌍 RDD를 입력받아, 같은 Key를 가진 모든 Value들을 하나의 리스트로 묶어 (Key, [Value List]) 형태의 새로운 RDD를 생성합니다.
    • 조인과 마찬가지로 클러스터 전체에 걸쳐 대규모 데이터 이동이 발생하므로 Wide Dependency를 생성합니다.

4. 연산 최적화의 핵심: Dependency 구조 이해하기

  • Spark가 분산 환경에서 작업을 빠르고 안정적으로 수행할 수 있는 이유는 DAG를 분석하여 연산 단계를 최적화하기 때문입니다. 이를 이해하기 위해서는 RDD 파티션 간의 의존성(Dependency)을 파악해야 합니다.

4.1 Narrow Dependency (빠름, 파이프라이닝 가능)

  • 정의: 자식 RDD의 각 파티션이 부모 RDD의 최대 1개 파티션에만 의존하는 경우를 말합니다.
  • 특징: 다른 노드와의 데이터 교환 없이, 하나의 워커 노드 메모리 안에서 데이터 흐름을 연속적으로 처리(Pipelining)할 수 있어 속도가 매우 빠릅니다.
  • 해당 연산: map, filter 등

4.2 Wide Dependency (느림, Shuffling 발생)

  • 정의: 하나의 부모 RDD 파티션 데이터가 여러 개의 자식 RDD 파티션으로 분할되어 의존 관계를 맺는 경우입니다.
  • 특징: 동일한 Key를 가진 데이터를 한 곳으로 모으기 위해 클러스터 노드 간에 네트워크를 통한 막대한 데이터 이동, 즉 셔플링(Shuffling)이 발생하므로 처리 속도가 크게 저하됩니다. 장애 발생 시 복구 비용도 큽니다.
  • 해당 연산: join, groupByKey 등

4.3 심화 Q&A: FlatMap의 의존성은 무엇일까?

  • FlatMap은 하나의 요소를 0개 혹은 여러 개로 만들 수 있으므로 혼동하기 쉽습니다.

  • Q1: FlatMap은 하나의 파티션으로부터 여러 개의 파티션을 생성할 수 있는가?

    • A: 논리적으로는 1:N(One-to-many) 함수이므로 가능합니다.

  • Q2: 그렇다면 데이터가 여러 곳으로 퍼지니 Wide Dependency인가?

    • A: 아닙니다. 비록 자식 RDD(R1)에서 여러 개의 파티션이 생성될지라도, 생성된 자식 파티션 각각의 입장에서 보면 자신의 출처는 오직 하나의 부모 파티션(R0)뿐입니다. 따라서 여전히 Narrow Dependency를 유지하며, 파이프라이닝이 가능합니다.

5. Pop Quiz: 분산 연산 최적화 (reduceByKey vs. groupByKey)

  • Spark에는 groupByKey 외에도 reduceByKey(func)라는 강력한 Transformation이 존재합니다. 이는 Key를 기준으로 그룹화를 하되, 그룹화 과정에서 사전에 정의된 reduce 함수 \((V, V) \Rightarrow V\)를 적용하여 값을 합쳐서 보냅니다.

문제: reduceByKey()를 사용하여 분산 데이터 상에서 가장 전형적인 “Word Count (단어 빈도수 세기)” 알고리즘을 구현해보세요.

  • 해결책 및 최적화 원리:
word_counts = sc.textFile(file) \
                .flatMap(lambda line: line.split()) \
                .map(lambda word: (word, 1)) \
                .reduceByKey(lambda a, b: a + b)
  • 왜 groupByKey().map(...) 대신 reduceByKey()를 써야 할까요?
    • groupByKey를 사용하면 ’단어’라는 Key를 기준으로 모든 1이라는 숫자들이 셔플링을 통해 네트워크를 이동한 뒤 하나의 큰 리스트([1, 1, 1, ..., 1])로 모입니다. 이는 심각한 네트워크 병목과 메모리 초과를 유발합니다.
    • 반면 reduceByKey를 사용하면 각 워커 노드에서 데이터를 네트워크로 보내기(Shuffling) 이전에 로컬에서 먼저 덧셈 연산(Local Reduce, Combiner 역할)을 수행합니다. 따라서 클러스터 네트워크를 타는 데이터의 크기가 극적으로 감소하여 분산 처리 성능이 비약적으로 향상됩니다.
